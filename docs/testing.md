# Testing Strategy & Coverage

This document explains the testing approach for both the Soroban smart contracts and the Next.js frontend, how to run tests locally, and how coverage is enforced in CI.

---

## Overview

| Layer | Framework | Location |
|---|---|---|
| Rust smart contracts | `cargo test` + `soroban_sdk::testutils` | `contracts/crowdfund/src/test.rs` |
| Frontend unit/integration | Jest + ts-jest + @testing-library/react | `apps/interface/src/**/*.test.{ts,tsx}` |
| Frontend E2E | Playwright | `apps/interface/` (playwright config) |

---

## Rust Contract Tests

### Structure

Contract tests live in `contracts/crowdfund/src/test.rs`, declared as a separate module from `lib.rs`:

```rust
// lib.rs
#[cfg(test)]
mod test;
```

The `#[cfg(test)]` gate ensures the test module is compiled only during `cargo test` and never included in the WASM output.

### Test Environment Setup

Every test uses `soroban_sdk::testutils` to create an in-process Soroban environment — no network or node required:

```rust
use soroban_sdk::{
    testutils::{Address as _, Ledger},
    Address, Env, String, Vec,
};
use crate::{CrowdfundContract, CrowdfundContractClient};

fn setup_contract(
    env: &Env,
    deadline: u64,
    goal: i128,
    min_contribution: i128,
) -> (Address, Address, CrowdfundContractClient<'_>, token::StellarAssetClient<'_>) {
    env.mock_all_auths(); // bypass auth checks for test simplicity

    let creator = Address::generate(env);
    let token_admin = Address::generate(env);
    let token_id = env.register_stellar_asset_contract(token_admin.clone());
    let token_admin_client = token::StellarAssetClient::new(env, &token_id);

    let contract_id = env.register_contract(None, CrowdfundContract);
    let client = CrowdfundContractClient::new(env, &contract_id);

    client.initialize(
        &creator, &token_id, &goal, &deadline, &min_contribution,
        &String::from_str(env, "My Title"),
        &String::from_str(env, "My Description"),
        &None, &None, &None,
    );

    (creator, token_id, client, token_admin_client)
}
```

Key testutils used:

- `Env::default()` — creates an isolated in-memory Soroban environment
- `env.mock_all_auths()` — bypasses `require_auth()` calls so tests focus on logic, not signatures
- `Address::generate(env)` — generates a random test address
- `env.register_stellar_asset_contract(admin)` — deploys a real SAC token for transfer testing
- `env.register_contract(None, CrowdfundContract)` — deploys the contract under test
- `env.ledger().set_timestamp(ts)` — manipulates ledger time to test deadline logic

### Example Tests

**Happy path — contribute and verify state:**

```rust
#[test]
fn initialize_and_contribute_updates_state() {
    let env = Env::default();
    let (_creator, token_id, client, token_admin_client) =
        setup_contract(&env, 1_000, 10_000, 100);

    let contributor = Address::generate(&env);
    token_admin_client.mint(&contributor, &500);
    client.contribute(&contributor, &500, &token_id);

    assert_eq!(client.total_raised(), 500);
    assert_eq!(client.contribution(&contributor), 500);
    assert!(client.is_contributor(&contributor));

    let stats = client.get_stats();
    assert_eq!(stats.contributor_count, 1);
    assert_eq!(stats.largest_contribution, 500);
}
```

**Error path — invalid platform fee rejected:**

```rust
#[test]
fn invalid_platform_fee_is_rejected() {
    let env = Env::default();
    env.mock_all_auths();
    // ...setup...
    let result = client.try_initialize(/* ... fee_bps: 10_001 ... */);
    assert_eq!(result.err(), Some(Ok(ContractError::InvalidFee)));
}
```

**Access control — token whitelist enforced:**

```rust
#[test]
fn accepted_token_whitelist_is_enforced() {
    // contribute with a non-whitelisted token
    let result = client.try_contribute(&contributor, &100, &other_token);
    assert_eq!(result.err(), Some(Ok(ContractError::TokenNotAccepted)));
}
```

Use `client.try_<method>()` (generated by `contractimpl`) to assert on `Err` variants without panicking.

### Running Contract Tests

```bash
# Run all workspace tests
cargo test --workspace

# Run only crowdfund contract tests
cargo test -p crowdfund

# Run a specific test by name
cargo test initialize_and_contribute
```

## Fuzz Testing (Proptest)

### Overview

Fuzz testing uses property-based testing to validate contract functions against a wide range of inputs, including edge cases and invalid values that traditional example-based tests might miss.

### Running Fuzz Tests

```bash
# Run all fuzz tests
cargo test --test fuzz_tests -- --nocapture

# Run with higher iteration count
cargo test --test fuzz_tests -- --proptest-config="cases=500"

# Run specific fuzz test
cargo test --test fuzz_tests fuzz_contribute_valid_amounts_succeeds
```

### Fuzz Test Categories

| Function | Test Coverage |
|---|---|
| `contribute` | Valid amounts, below-min rejection, max limit enforcement, large contributor counts |
| `refund_single` | Post-cancellation refund, post-deadline refund, double-claim prevention |
| `withdraw` | Successful campaign withdrawal, platform fee calculation, vesting schedules |
| Edge cases | Overflow protection, negative amounts, concurrent operations |

### Adding New Fuzz Tests

Add new tests to `contracts/crowdfund/tests/fuzz_tests.rs`:

```rust
proptest! {
    #![proptest_config(ProptestConfig::with_cases(200))]

    #[test]
    fn my_fuzz_test(value in any_valid_input_strategy()) {
        // Test implementation with assertions
    }
}
```

## E2E Tests (Playwright)

Playwright tests cover critical user flows end-to-end against a running Next.js dev server.

### Running E2E Tests Locally

```bash
# Install browsers (first time only)
npx playwright install --with-deps

# Run all E2E tests
npm run test:e2e
```

On failure, Playwright saves a report to `playwright-report/`. Open it with:

```bash
npx playwright show-report
```

## CI/CD Testing Workflows

### `rust_ci.yml` — triggered on push/PR to `main`

1. Checks out code and installs the stable Rust toolchain with the `wasm32-unknown-unknown` target
2. Builds the contract to WASM (`cargo build --release --target wasm32-unknown-unknown`)
3. Asserts the WASM binary is under the 64 KB Soroban size limit
4. Runs all Rust tests (`cargo test --workspace`)

### `frontend_ci.yml` — triggered on push/PR to `main`

1. Sets up Node.js 20 with `node_modules` caching
2. Runs ESLint across the workspace (`npm run lint`)
3. Runs TypeScript type-checking (`tsc --noEmit`)
4. Runs `npm audit --audit-level=high` to catch high-severity dependency vulnerabilities

> Note: Jest tests are run locally and coverage is enforced via `npm run test:coverage`. The CI workflow focuses on static analysis; add a `npm test` step to `frontend_ci.yml` to gate merges on passing unit tests.

### `playwright.yml` — triggered on PRs targeting `main`

1. Sets up Node.js 20 with caching
2. Installs Playwright browsers and system dependencies
3. Runs the full E2E suite (`npm run test:e2e`)
4. On failure, uploads the Playwright HTML report as a CI artifact (retained for 7 days)

### `deploy-testnet.yml` — triggered on push to `develop`

Deploys contracts to Stellar testnet. Not part of the test suite but validates that the built WASM deploys and initializes correctly against a live network.
